# Guru UI â€” Design Plan

## What it is

A standalone web app that reads the Vedam vault directory (live, locally) and renders it as an interactive graph â€” deployable to any static host with a build step.

---

## Stack

| Layer | Choice | Why |
|---|---|---|
| Build tool | Vite + React | Fast dev server, easy static build |
| Graph | `react-force-graph` | Wraps D3 force simulation, handles large graphs, 2D/3D toggle built-in |
| Markdown + LaTeX | `unified` + `remark-math` + `rehype-katex` | Handles `$...$` and `$$...$$` which Guru generates throughout |
| Wikilinks | Custom remark plugin | Converts `[[note title]]` â†’ clickable in-app navigation |
| Local server | Node.js + Express | Reads live vault files from `config.json` path |
| Routing | `react-router-dom` | Client-side routing for `/note/:title` and `/domain/:parentDomain` |
| Fuzzy search | `fuse.js` | Client-side fuzzy match on note titles; no backend dependency |
| Styling | Tailwind CSS | Minimal setup, dark mode native |

---

## Two Modes

**Local mode (`npm run dev`)**
- Express server reads vault files live from `vault_path` in `config.json`
- On startup, `fs.accessSync` checks that `vault_path` exists â€” if not, prints `vault_path not found: <path>` and exits with code 1
- Watches for filesystem changes, pushes updates over WebSocket
- Graph re-renders automatically when notes are added/approved
- If the WebSocket connection drops, the React client reconnects with exponential backoff (1s â†’ 2s â†’ 4s, capped at 30s) and shows a "Disconnected â€” reconnecting..." banner

**Deploy mode (`npm run build` â†’ static host)**
- `npm run build:vault` runs `scripts/build-vault.js`, which reads the vault and writes `public/vault.json`; sets `VITE_DEPLOY=true` for the subsequent Vite build
- The React app detects `import.meta.env.VITE_DEPLOY` at build time: if true, `useVault.js` fetches `/vault.json`; otherwise it fetches `/api/graph`
- Deploy the `dist/` folder to Vercel, GitHub Pages, or any CDN
- To update: re-run the build script after approving new notes, redeploy

**Failure modes**
- `config.json` missing at startup â†’ Express exits immediately with a readable error; do not start
- `vault_path` unreachable at startup â†’ same: exit with path in error message
- `/api/graph` or `/vault.json` fetch fails in React â†’ `useVault` returns `{ error }`, Graph renders a full-screen error message
- Individual note fetch fails â†’ `useNote` returns `{ error }`, NotePanel renders an inline error instead of content

## Data Model

The parser extracts this structure from every `.md` file:

```js
// Node
{
  id: "probability bayes theorem",       // = frontmatter title
  title: "probability bayes theorem",
  domain: "probability",
  parentDomain: "mathematics",
  source: "...",
  filePath: "probability/probability bayes theorem.md",
  isBridge: boolean,    // true if note bridges â‰Ą2 distinct domains
  content: "...",       // full markdown body (stripped of frontmatter)
}

// Edge
{
  source: "probability bayes theorem",
  target: "probability conditional probability",
  type: "prereq" | "builds-into" | "related"
}
```

**Field normalization:** frontmatter keys are mapped to camelCase in the parsed node â€” `parent-domain` â†’ `parentDomain`. All other keys are unchanged.

**`filePath`** is relative to `vault_path` in `config.json`. The Express server resolves it to an absolute path when serving `/api/note/:title`.

**Bridge detection** runs as a post-processing pass (`resolveGraph(nodes, edges)`) after all files are parsed â€” `isBridge` cannot be set per-file because the second condition requires the full edge set. A note is flagged as a bridge if:
1. Its `## Bridge to other domains` section (case-insensitive match) contains 2+ strings that match known `domain` values in the parsed node set, or
2. It appears in `builds-into` edges originating from notes in 2+ distinct domains

**Wikilink parsing:** strip `[[` and `]]` from `prereqs`, `builds-into`, `related` frontmatter arrays â†’ edge list.

## Graph View

**Nodes:**
- Color = `parent-domain` (one color per top-level category, legend shown)
- Size = degree (number of edges) â€” bridge concepts naturally become larger
- Shape = circle normally, diamond for bridge concepts
- Glow/outline on hover

**Edges:**

| Type | Style | Direction | Arrow convention |
|---|---|---|---|
| `prereq` | solid line | directed | dependent â†’ dependency (A needs B: A â†’ B) |
| `builds-into` | dashed line | directed | foundation â†’ advanced concept |
| `related` | dotted line, lower opacity | undirected | none |

Use `linkDirectionalArrowLength` in `react-force-graph` to render arrows on directed edges.

**Default state:** `related` edges are hidden on initial load (the edge type toggle starts unchecked for `related`). This reduces visual clutter for a dense vault.

**Interactions:**
- Click node â†’ opens NotePanel (right drawer slides in)
- Hover node â†’ Tooltip shows title + domain + edge count
- Click a wikilink inside NotePanel â†’ calls `router.push('/note/:title')`, navigates graph to that node and opens its note; browser back button works
- Double-click canvas background â†’ deselect, close panel

## Filter Bar (Left Panel)

- **Search:** text input, fuzzy match on `title` field via `fuse.js` (threshold 0.3) â†’ highlights matching nodes, dims others
- **Parent-domain toggles:** one pill per `parent-domain`, click to show/hide that cluster
- **Bridge only toggle:** dim all non-bridge nodes
- **Edge type toggles:** show/hide prereq / builds-into / related edges independently
- **Depth slider:** when a node is selected, show only nodes within N hops

---

## Note Panel (Right Drawer)

Renders the full note markdown with:
- Frontmatter displayed as a metadata card (title, domain, source) â€” not as raw YAML
- `$...$` and `$$...$$` rendered by KaTeX
- `[[wikilink]]` rendered as clickable in-app links
- Section headings as a mini table-of-contents at top
- Backlinks section at bottom (auto-computed: which other notes link to this one)

## URL Routing

```
/                      â†’ graph (full vault)
/note/:title           â†’ graph focused on that node + panel open (shareable link)
/domain/:parentDomain  â†’ graph filtered to that parent-domain cluster
```

React Router handles this. In local mode, Express serves `index.html` for all routes. In deploy mode, configure the host to redirect 404s to `index.html`.

## Local â†’ Deploy Path

```bash
# One-time setup
cd guru-ui
ln -sf ../config.json config.json   # symlink to GURU-AI root config
npm install

# Local development (live vault)
npm run dev        # starts Express on :3001 + Vite on :5173

# Deploy build
npm run build:vault   # runs scripts/build-vault.js â†’ writes public/vault.json
npm run build         # Vite builds dist/ with VITE_DEPLOY=true
# deploy dist/ to Vercel / GitHub Pages / S3
```

`package.json` build script:
```json
"build": "VITE_DEPLOY=true vite build"
```

`build-vault.js` is the only step that needs access to the vault directory. After that, `dist/` is a fully self-contained static site.

**`build-vault.js` error handling:** on malformed frontmatter or unreadable file, skip that file, log `[WARN] Skipped: <filename> â€” <reason>` to stderr, and continue. Print a skipped-file count in the summary line at the end. Do not abort the build for a single bad note.

**Express logging:** use `morgan` for request logging in dev. Vault path errors use `console.error` + `process.exit(1)`.

## Key Design Constraints

- **No database.** Graph data is derived entirely from `.md` frontmatter at parse time.
- **No authentication.** Read-only viewer. Writes still go through Claude Code + Guru.
- **Vault is source of truth.** The UI never modifies vault files.
- **Config reuse.** `guru-ui/config.json` points at the same `vault_path` as the Guru pipeline â€” same file, no duplication.
